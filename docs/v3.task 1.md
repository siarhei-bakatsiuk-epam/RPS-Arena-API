## 1. Общее описание

**RPS Arena** — платформа для проведения онлайн-турниров по игре «камень-ножницы-бумага». Игроки регистрируются, играют матчи друг против друга (каждый матч состоит из нескольких раундов), а платформа ведёт учёт результатов и формирует турнирную таблицу в реальном времени.

Система состоит из двух сервисов:

- **MatchService** — управляет игроками и матчами (CRUD + поиск)
- **LeaderboardService** — асинхронно обрабатывает результаты матчей и формирует рейтинговую таблицу

## 3. MatchService

### 3.1 Управление пользователями

Сервис предоставляет API для работы с учётными записями игроков: регистрации, получения профиля (по ID), просмотра списка всех игроков, обновления данных и удаления. Конкретные маршруты и HTTP-методы — на усмотрение разработчика.

**Модель пользователя:**
```json
{
  "id": "uuid",
  "username": "string",      // уникальный
  "email": "string",         // уникальный
  "createdAt": "datetime"
}
```

### 3.2 Управление матчами

Сервис предоставляет API для записи результатов сыгранных матчей, получения матча по ID, просмотра списка матчей (с фильтрацией) и истории конкретного игрока. Конкретные маршруты и HTTP-методы — на усмотрение разработчика.

**Модель матча:**
```json
{
  "id": "uuid",
  "playerOneId": "uuid",
  "playerTwoId": "uuid",
  "playerOneScore": 3,
  "playerTwoScore": 1,
  "playedAt": "datetime"
}
```

> `playerOneScore` / `playerTwoScore` — количество выигранных раундов каждым игроком в рамках матча.

**Правила:**
- Оба игрока должны существовать
- Игрок не может играть сам с собой
- Оба счёта — неотрицательные целые числа
- Ничья допустима (счёт `2:2`, `0:0` и т.д.)
- После сохранения матча публикуется событие `MatchRecorded` для LeaderboardService

**Фильтрация списка матчей:**
- `?playerId=uuid` — матчи с участием игрока
- `?from=datetime&to=datetime` — по дате
- `?page=1&pageSize=20` — пагинация

## 4. LeaderboardService

Самостоятельный сервис, подписанный на события от MatchService.

### 4.1 Обработка событий

При получении `MatchRecorded` сервис пересчитывает статистику игрока по двум осям:

**По итогам матчей:**

| Результат | Очки матча |
|-----------|-----------|
| Победа    | 3         |
| Ничья     | 1         |
| Поражение | 0         |

**По счёту в матчах:**
- `totalScore` — суммарное количество выигранных раундов игроком по всем матчам

*(Схему начисления очков за победу/ничью/поражение можно скорректировать — обоснование приветствуется.)*

### 4.2 API турнирной таблицы

Сервис предоставляет эндпоинт для получения топа игроков и эндпоинт для получения статистики конкретного игрока. Конкретные маршруты — на усмотрение разработчика.

**Параметры эндпоинта списка лидеров** *(маршрут и структура могут отличаться)*:
- `?sortBy=wins|draws|losses|matchPoints|totalScore` — критерий сортировки
- `?top=10` — количество записей (по умолчанию 10)

**Модель статистики игрока:**
```json
{
  "playerId": "uuid",
  "username": "string",
  "wins": 15,
  "losses": 4,
  "draws": 2,
  "totalMatches": 21,
  "matchPoints": 47,
  "totalScore": 63,
  "rank": 1
}
```

> `matchPoints` — турнирные очки (3/1/0 за победу/ничью/поражение)  
> `totalScore` — сумма выигранных раундов по всем матчам

> Модели являются рекомендацией. Изменения в структуре данных приветствуются при наличии обоснования.

## 5. Защита от коллизий

1. **Идемпотентность записи матча** — повторный запрос с теми же данными не создаёт дубликат.

2. **Race condition при обновлении статистики** — параллельные события на одного игрока обрабатываются корректно.

## 6. Инфраструктура (Docker Compose)

Файл `docker-compose.yml` поднимает:

```
├── matchservice        — MatchService API
├── leaderboardservice  — LeaderboardService
├── postgres            — БД (при необходимости экземпляров может быть больше одного)
└── rabbitmq            — брокер сообщений
```

Требования:
- Все сервисы стартуют командой `docker-compose up --build`
- Миграции EF Core применяются автоматически при старте (через `dotnet ef database update` в entrypoint или отдельный init-контейнер)
- Порты пробрасываются наружу для ручного тестирования (Swagger доступен)

---

## 7. Архитектурные требования

- **CQRS**: команды (Commands) и запросы (Queries) разделены; каждый обработчик — отдельный класс
- **MediatR**: все команды и запросы проходят через `IMediator`
- **FluentValidation**: каждый запрос/команда имеет свой `AbstractValidator<T>`, подключённый через pipeline behaviour
- **Миграции БД**: все изменения схемы управляются исключительно через EF Core Migrations; прямое редактирование схемы не допускается
- **Слоистая архитектура**: рекомендуется `Infrastructure → Domain ← Application ← API`
